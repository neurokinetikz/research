# v4 Plan: FOOOF Parameter Optimization

**Date:** 2026-04-11
**Status:** Draft -- to be refined after v3 results
**Goal:** Find the FOOOF parameter region where enrichment profiles converge, separating robust biological signal from parameter-sensitive noise

---

## Background

Through v1-v3, we fixed structural issues (f₀ mismatch, Hz-correction, merged theta+alpha, bandwidth floor, 50 Hz notch). What remains is parameter sensitivity -- the enrichment values depend on FOOOF settings that were chosen by convention, not optimization:

- `peak_threshold=0.001` (extremely permissive; FOOOF default is 2.0)
- `max_n_peaks=15` (principled but untested against alternatives)
- `min_peak_height=0.0001` (essentially no floor)
- `nperseg` (adaptive per band; gamma gets coarse resolution)

The v2 analysis showed that power filtering at analysis time partially compensates for permissive extraction. But we don't know if stricter extraction would produce different (better?) results, or if the analysis-time filter is sufficient.

## Parameter Grid

### Extraction parameters (require re-extraction)

| Parameter | Values | Current v3 | Unit | Rationale |
|---|---|---|---|---|
| peak_threshold | 0.001, 0.5, 1.0, 2.0 | 0.001 | Standard deviations of residual | Relative filter: how many SD above noise a bump must be. FOOOF default = 2.0. Our v3 = 0.001 (everything). |
| min_peak_height | 0.0001, 0.05, 0.10, 0.20 | 0.0001 | Absolute log10 power | Absolute filter: minimum height above aperiodic regardless of noise. 0.05 = ~25th percentile of gamma peaks. 0.10 = ~median of gamma. 0.20 = ~75th of gamma / ~median of beta-low. |
| max_n_peaks | 5, 10, 15, 20 | 15 | Count per FOOOF fit | How many peaks per channel per band. |
| nperseg_floor | adaptive, 2500 | adaptive | Samples | Resolution and min bandwidth for beta-high/gamma. |

**Interaction structure:**
- `peak_threshold` × `min_peak_height`: Both filter peaks but on different axes (relative vs absolute). Most interesting interaction -- does combining a moderate relative threshold (1.0 SD) with a moderate absolute floor (0.05) give cleaner results than either alone?
- `peak_threshold` × `max_n_peaks`: With strict threshold, fewer peaks found, so max_n_peaks matters less. With lenient threshold, max_n_peaks becomes the binding constraint.
- `nperseg_floor` is mostly independent of the others -- it affects which peaks CAN be detected (resolution), not which peaks survive the threshold.

## What We Measure

### Primary metric: Enrichment profile stability

For each band, compute the 13-position enrichment profile. Across the 384 analysis configurations:

1. **Profile correlation matrix** -- r between every pair of configurations. The "stable core" is the set of configurations with mutual r > 0.95.
2. **Position-level CV** -- coefficient of variation for each position across configs. Positions with low CV are robust; positions with high CV are parameter-sensitive.
3. **Sign stability** -- how many configs agree on the sign (>+5% or <-5%) at each position.

### Secondary metrics

4. **Peak count** -- how many peaks survive as a function of threshold × max_peaks
5. **Aperiodic fit quality** -- does stricter peak detection improve R² (suggesting FOOOF was overfitting noise)?
6. **Bandwidth distribution** -- does the distribution change with threshold (are the rejected peaks systematically broad or narrow)?

## Key Questions

### Q1: Does peak_threshold matter after power filtering?

If enrichment profiles are the same at threshold=0.001+filter=50% vs threshold=2.0+filter=0%, then the extraction threshold is redundant with analysis-time filtering. This would mean our v3 approach (permissive extraction, filter at analysis) is optimal.

If they differ, the extraction threshold affects the APERIODIC FIT, which power filtering can't undo. Higher threshold → fewer peaks modeled → different aperiodic → different peak detection at other positions.

**Prediction:** They'll differ slightly. threshold=1.0+filter=25% will approximate threshold=0.001+filter=50%, but not perfectly.

### Q2: Is there a max_n_peaks sweet spot?

Too low (5): misses real peaks, biases toward dominant frequency. 
Too high (20): FOOOF overfits noise, distorts aperiodic.
We found cap=15 doesn't hit the limit (max per channel was 13 in gamma).

**Prediction:** max_n_peaks ≥ 12 all give the same results. Below 12, enrichment concentrates artificially at the tallest-peak positions.

### Q3: Does the nperseg floor help gamma?

With floor=2500, gamma gets 0.100 Hz resolution instead of 0.212 Hz. This allows narrower peak detection and better bandwidth discrimination.

**Prediction:** Gamma enrichment profiles are similar at both resolutions. Bandwidth distributions differ (more narrow peaks detected at finer resolution).

### Q4: What's the optimal power filter?

The v2 Dortmund sweep showed signal strengthening from 0% to 90% with no plateau. Is this the same across all FOOOF parameter settings?

**Prediction:** The optimal filter depends on peak_threshold. With threshold=0.001, filter=50% is needed. With threshold=2.0, filter=0% might be sufficient (strict threshold already rejects weak peaks).

### Q5: Does R² filtering add value?

v2 showed R² barely mattered (profiles stable from 0.70 to 0.95). Is this still true across different peak_threshold settings?

**Prediction:** Still doesn't matter. R² reflects overall fit quality, not individual peak quality.

## Execution Plan

### Phase 1: Extraction grid (~1 hour)

```bash
for thresh in 0.001 0.5 1.0 2.0; do
    for maxp in 5 10 15 20; do
        for nperseg_floor in adaptive 2500; do
            python scripts/run_v4_sweep.py --dataset eegmmidb \
                --peak-threshold $thresh \
                --max-n-peaks $maxp \
                --nperseg-floor $nperseg_floor
        done
    done
done
```

Output: 32 directories in `exports_v4_sweep/`, each containing 109 subject CSVs.

### Phase 2: Analysis grid (~10 min)

For each of the 32 extractions, compute enrichment at 4 power filters × 3 R² thresholds = 12 analyses. Total: 384 enrichment profiles.

### Phase 3: Stability analysis (~5 min)

- Compute 384×384 profile correlation matrix
- Identify the convergent parameter region
- Compute position-level stability metrics
- Generate heatmaps of enrichment × parameter space

### Phase 4: Validation

- Extract Dortmund and LEMON with optimal parameters
- Compare enrichment and cognitive correlations to v3
- If improvements: full 17-condition extraction
- If no improvement: v3 parameters are sufficient

## What Would Change the Plan

**From v3 results:**
- If bandwidth filtering adds significant value → prioritize nperseg_floor=2500
- If power filter shows a plateau at 50% → narrow the filter grid to 25-75%
- If v3 enrichment already matches v2 closely → the parameter sweep may not be worth it
- If LEMON subject count issue reveals a data problem → fix before v4

**From v4 Phase 1-3:**
- If all 384 profiles are nearly identical → parameters don't matter, v3 is fine
- If there are two distinct clusters → need to understand what separates them
- If one parameter dominates → focus on that parameter, fix others
